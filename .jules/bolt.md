## 2024-05-18 - Throttling Scroll and Debouncing Search in Jekyll index.html
**Learning:** In a static site architecture like Jekyll where all content is rendered into a single HTML file and filtered via client-side JavaScript, synchronous DOM manipulations tied to high-frequency events (`scroll` and `input`) create severe main-thread bottlenecks, especially on long pages with many elements (like a large paper index).
**Action:** Always wrap `scroll` event listeners in `requestAnimationFrame` with a ticking flag, and implement caching (e.g., `lastActive`) to only write to the DOM when state actually changes. Always wrap `input` event listeners for text filtering in a `setTimeout` debounce to delay expensive DOM updates until the user pauses typing.
## 2025-02-27 - Caching Active Index and Throttling Scroll in Paper TOC
**Learning:** In the `paper.html` layout, the Table of Contents (TOC) link highlighting previously executed layout calculations (`offsetTop`) and un-cached DOM updates (`classList.toggle`) synchronously on every scroll tick. This continuous re-rendering caused main-thread blocking, particularly detrimental for long paper notebooks.
**Action:** Always employ an active-state caching variable (e.g., `lastActive`) alongside a `requestAnimationFrame` throttle with a ticking flag for scroll event listeners to limit DOM manipulation solely to state changes.
## 2025-05-06 - Preventing Unnecessary DOM Mutations in Client-Side Filters
**Learning:** Assigning a value to `el.innerHTML`, even if the new string is identical to the existing `innerHTML`, causes the browser to destroy and recreate the DOM nodes within that element. This triggers expensive style recalculations and layout thrashing, which is particularly detrimental when performed in loops over many elements (like processing KaTeX math across all headings, paragraphs, and list items).
**Action:** Always diff-check strings before mutating the DOM. Use `if (newHtml !== html) { el.innerHTML = newHtml; }` instead of unconditionally assigning it. Additionally, add early returns using simple string searches (e.g., `indexOf`) before executing expensive Regular Expressions if the presence of target substrings is a prerequisite for replacement.
## 2025-05-08 - Caching DOM Structure and Attributes for Client-Side Filtering
**Learning:** In client-side filtering logic (like the paper search in `index.html`), querying the DOM repeatedly via `querySelectorAll` and extracting attributes via `getAttribute` on every input tick is a massive bottleneck. These synchronous DOM reads block the main thread and cause noticeable jank on large pages, even when wrapped in a debounce.
**Action:** Always pre-compute and cache the DOM structure (nodes and hierarchy) along with normalized attributes (e.g., lowercase search text) into an array or object on page load or outside the event listener. Filtering logic should solely iterate over these fast, in-memory representations and only write to `style.display` when updating the view.
## 2025-05-08 - Short-circuiting Expensive Operations in String Processing
**Learning:** In text-processing utility functions (`scripts/_common.py` `is_stub`), unconditionally executing expensive multi-line regular expressions (`re.search(..., re.MULTILINE)`) over large strings is a major performance bottleneck, especially when iterated over hundreds of files during site builds (`prepare_pages.py`).
**Action:** Always optimize heavy string operations by restructuring checks sequentially from least-to-most expensive: First, check conditions that allow early returns (e.g., character counts). Second, use fast, native substring checks (`'substring' in string`) as a strict prerequisite before falling back to complex regular expressions.
## 2025-05-18 - Avoid creating copies of large strings
**Learning:** In Python performance optimizations, using `.lower()` on a large text body (e.g., markdown content) just for a case-insensitive substring check creates a full memory copy, which can outweigh the performance benefit of avoiding a regular expression.
**Action:** Always check explicitly for common exact case variations (e.g., `if 'arxiv' not in content and 'arXiv' not in content:`) instead of calling `.lower()` on the entire string when implementing short-circuit checks.
## 2025-05-18 - Eliminating Layout Thrashing in Scroll Spies
**Learning:** While `requestAnimationFrame` and state caching prevent unnecessary DOM *writes* during scroll events, performing DOM *reads* for layout metrics like `offsetTop` or `getBoundingClientRect()` inside a scroll handler still causes severe performance penalties. The browser is forced to synchronously recalculate layout on every scroll tick to return these values, defeating much of the benefit of throttling.
**Action:** Always pre-compute and cache DOM layout metrics into an array or object. In the scroll handler, only perform mathematical comparisons against `window.scrollY` and the cached array. Update the cached layout positions asynchronously using `ResizeObserver` (or debounced `resize` events) to ensure accuracy when the page layout changes.
## 2025-05-18 - Skipping Irrelevant Nodes in DOM Traversal
**Learning:** In KaTeX math rendering within `_layouts/default.html`, executing expensive regular expressions on every text node indiscriminately is a significant bottleneck. Large blocks of code (like those inside `<pre>` or `<code>` tags) are evaluated despite inherently not being targeted by the math engine. Furthermore, without a flag (`window.katexDone`), KaTeX re-scans the entire DOM on every check interval until `renderMathInElement` successfully triggers, wasting resources.
**Action:** Always inject a `NodeFilter` when using a `TreeWalker` to proactively skip irrelevant sub-trees (such as `SCRIPT`, `STYLE`, `PRE`, and `CODE`).
## 2025-10-24 - Optimizing Regex with Common String Variant Pre-checks
**Learning:** In Python, case-insensitive regular expression searches (`re.search(..., re.IGNORECASE)`) are exceptionally slow on large strings, even if they return early (due to overhead). Conversely, Python's `in` string operator is highly optimized in C and very fast. However, replacing a case-insensitive regex completely with an `in` check requires specifying *all* permutations (e.g. 32 variants for "arxiv"), which is impractical and error-prone.
**Action:** When optimizing a regex that acts as a short-circuit guard on large strings, use the `in` operator to check for the most *common* casing variants (e.g., `'arxiv'`, `'arXiv'`) as the first-pass fast path, but always keep the original `re.search` as a fallback. This achieves dramatic speedups for the vast majority of cases without introducing subtle correctness regressions.
## 2025-10-25 - O(1) Fast-Path Lookups for Fuzzy Title Matching
**Learning:** In metadata synchronization scripts like `prepare_pages.py` and `sync_progress.py`, falling back immediately to O(N) fuzzy string matching (e.g., checking if one title is a substring of another across all known titles) causes significant performance degradation as the dataset grows. When there are hundreds or thousands of elements, this creates an O(N*M) bottleneck during processing, where M is the number of notes and N is the number of known titles.
**Action:** Always attempt an exact, O(1) dictionary lookup (using `dict.get()`) using the normalized title or directory name *before* executing the O(N) loop for fuzzy matching. This resolves the vast majority of matches instantly, keeping the expensive fuzzy matching strictly as a fallback for renamed or loosely correlated files.
## 2025-10-25 - Using parse_frontmatter to Extract Metadata Optimally
**Learning:** In metadata synchronization scripts like `prepare_pages.py`, executing multi-line regular expressions (`re.search(..., re.MULTILINE)`) across the entirety of large markdown files to extract YAML frontmatter values is a significant performance bottleneck. When run in a loop over hundreds of files, these unanchored regexes exhibit O(N) scaling relative to the file size.
**Action:** Always utilize the existing fast-path `parse_frontmatter` utility (from `_common.py`) to parse the YAML block directly into a dictionary instead of scanning the full file content with regular expressions. Extract required metadata properties via simple, O(1) dictionary lookups (`dict.get()`).
## 2024-05-16 - Optimize markdown file parsing to avoid large string copies
**Learning:** Using `string.split('---', 2)` or array slicing to skip frontmatter in large markdown files triggers full memory copies of the entire file body. In Python, this becomes a significant O(N) memory allocation and performance bottleneck when processing hundreds of large notes. Similarly, running `re.search` over the entire document string when you only need to match a header is unnecessarily slow.
**Action:** Always use exact `string.find()` index lookups coupled with `string[start:end]` substring views (which are much smaller) for parsing localized sections (like frontmatter). Additionally, use `string.find('#') != -1` as an extremely fast O(N) pre-check before executing complex O(N) regex searches on large bodies of text.
## 2025-10-25 - Using native str.count over re.findall for counting distinct characters
**Learning:** In Python, utilizing `re.findall(r'[✅📖⏳]', text)` to count the occurrences of specific emojis/characters inside a large markdown file is significantly slower than chaining native string count methods (`text.count('✅') + text.count('📖') + text.count('⏳')`). The overhead of the regular expression engine is much higher than multiple fast, C-optimized string scans for individual characters.
**Action:** When counting occurrences of a small set of exact, distinct strings or characters, always prefer summing the results of `str.count()` over using a regex character class with `re.findall()`.
## 2025-10-25 - Efficient string whitespace squashing and alphanumeric filtering
**Learning:** In string normalization routines (like `normalize_name`), calling `re.sub(r'\s+', ' ', name).strip()` to squash multiple spaces and trim boundaries is significantly slower than the native Python `split()` followed by `' '.join()`. Furthermore, uncompiled `re.sub` calls in tight loops introduce unnecessary regex parsing overhead.
**Action:** When filtering out characters and normalizing whitespace, compile the `re.sub` pattern for the character exclusion globally, and rely on `' '.join(filtered_string.split())` for the whitespace squashing rather than a secondary regex. This yields a >2x performance improvement.
